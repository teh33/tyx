# `tyx.lock` MVP semantics

`tyx.lock` is Tyx's committed environment snapshot for the MVP. It is generated by `tyx up` from `project.tyx` plus safe local checks.

The lockfile is intentionally honest: it records what Tyx requested and observed, but it does **not** claim full Nix-style hermetic reproducibility yet.

## Commit policy

Commit both files:

- `project.tyx` — desired project environment
- `tyx.lock` — last resolved/checked environment snapshot

For the MVP, `tyx.lock` may contain host-observed state such as whether tools were present on `PATH` or whether `node_modules` existed. This is useful for agents and humans reviewing setup, but it is not a guarantee that another machine has the same state.

A future Tyx may split this into:

- committed reproducibility lock
- local machine state under `.tyx/state`

Do not overstate the MVP lockfile as hermetic.

## Format

`tyx.lock` uses the same tiny line-oriented style as `project.tyx`.

Current header:

```tyx
# Generated by tyx. Commit this file with project.tyx.
lock 1
```

The `lock 1` line is a schema/version marker, not a section header.

## Sections

### `tools:`

```tyx
tools:
node requested 22 status present version v22.13.1 matches true provider path
pnpm requested 9.15.0 status missing provider path
```

Fields:

- tool name
- `requested <version>` from `project.tyx`
- `status present|missing|unsupported`
- optional `version <observed-version>`
- optional `matches true|false` for present tools
- `provider path` for current PATH-based detection

### `services:`

```tyx
services:
compose file compose.yaml status present
compose file compose.override.yaml status missing
```

MVP services only record compose file presence. Tyx does not start Compose services yet.

### `env:`

```tyx
env:
example .env.example status present
file .env status missing
```

Tyx records env file presence without reading secrets as config.

### `dependencies:`

```tyx
dependencies:
pnpm manifest package.json path node_modules status missing
```

MVP dependency checks are intentionally simple: for Node/TypeScript script runners, Tyx checks `package.json` and `node_modules`. It does not run package installs yet.

### `scripts <runner>:`

```tyx
scripts pnpm:
dev
test
"dev server"
```

Scripts are copied from the parsed project config so lock readers can see the commands Tyx can resolve.

## Reproducibility level

MVP `tyx.lock` provides:

- requested tool versions
- observed local tool versions/status
- file presence status
- dependency presence status
- script runner mapping

MVP `tyx.lock` does not provide:

- content-addressed package graph
- downloaded binary hashes
- Docker image digests
- package-manager lock normalization
- rollback guarantees
- cross-machine hermeticity

This is acceptable for Tyx Dev's first milestone: repo setup should be inspectable, repeatable enough for local onboarding, and agent-friendly.
