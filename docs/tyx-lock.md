# `tyx.lock` MVP metadata

`tyx.lock` is Tyx's resolved setup metadata for the MVP. It is written by `tyx up` from `project.tyx` plus safe local checks and delegated setup commands.

The lockfile is intentionally honest: it records what Tyx requested and observed, but it does **not** claim full Nix-style hermetic reproducibility.

## Commit policy

Commit both files:

- `project.tyx` — desired project environment
- `tyx.lock` — last resolved/checked setup snapshot

For the MVP, `tyx.lock` may contain host-observed state such as whether tools were present on `PATH` or whether `node_modules` existed. This is useful for agents and humans reviewing setup, but it is not a guarantee that another machine has the same state.

## Format

`tyx.lock` uses Tyx's tiny line-oriented style.

Header:

```tyx
# Generated by tyx. Commit this file with project.tyx.
lock 1
```

The `lock 1` line is a schema/version marker.

## Current sections

### `tools:`

```tyx
tools:

```

### `services:`

```tyx
services:
compose file compose.yaml status present

```

### `env:`

```tyx
env:
example .env.example status present
file .env status missing
```

### `dependencies:`

```tyx
dependencies:
pnpm manifest package.json path node_modules status present
```

### `scripts <runner>:`

```tyx
scripts pnpm:
dev
test
"dev server"
```

## Safety rules

- Do not include secrets, env values, access tokens, full process environments, or machine-unique identifiers.
- Record delegated setup state plainly; do not overstate reproducibility.
- Keep output deterministic enough for fixture tests and agent review.
- Increment the lock schema number for incompatible format changes.
